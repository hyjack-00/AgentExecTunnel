# Evaluation v0.3.2

## Changes

### executor / config

- `agent_exec_tunnel/config.py` — new `Settings.executor_shell` (default `/bin/bash` on POSIX, Git Bash on Windows with fallback to `cmd.exe`) and `Settings.executor_shell_args` (default `["-c"]`, or `["/c"]` for `cmd.exe`). Env overrides `AET_EXECUTOR_SHELL` / `AET_EXECUTOR_SHELL_ARGS`. Uses `field(default_factory=…)` for lazy evaluation.
- `agent_exec_tunnel/executor.py` — `Popen(task["command"], shell=True)` → `Popen([cfg.executor_shell, *cfg.executor_shell_args, task["command"]], shell=False)`. This is the whole point of the release: Python's `shell=True` hardcodes `/bin/sh` / `cmd.exe` as a wrapper layer; shell=False + explicit argv bypasses it.
- `VERSION`: `v0.3.1 → v0.3.2`. `PACKAGE_VERSION`: same.

### submitter

- `submitter/submit.py` — **new**, bottom-of-stack CLI. Preview: `-> <executor_shell> -c '<payload>'` / `  -> <payload>`. Submits raw payload, no rendering.
- `submitter/submit_gitbash.py` — **reverted** v0.3.1's client-side `<git bash>` cmdline wrap; now submits raw payload. Preview unchanged (still shows the intent-level `"<git bash>" -c <payload>` form as a human convenience).
- `submitter/submit_gitbash_ssh.py` — **reverted** v0.3.1's outer cmdline wrap; now submits just `relay_script` (the ssh base64 trampoline). Preview unchanged.
- `submitter/submit_powershell.py`, `submit_powershell_ssh.py`, `submit_bash.py` — unchanged from v0.3.1.

### tests

- `tests/test_executor_flow.py::ExecutorShellConfigTests::test_executor_popen_uses_configured_shell_with_shell_false` — **new**: mocks `subprocess.Popen`, asserts v0.3.2 invocation shape.
- `tests/test_cli_entrypoints.py::test_submit_plain_submits_raw_payload` — **new**: covers `submit.py`.
- `tests/test_cli_entrypoints.py::test_submit_gitbash_main_submits_raw_payload` — **rewritten** to assert raw-payload submission (was "wrapped cmdline" in v0.3.1).
- `tests/test_cli_entrypoints.py::test_submit_gitbash_ssh_main_submits_base64_relay_script` — **rewritten** to assert the trampoline string without outer cmdline wrap.

Full suite: **61 passing, 0 failing** (was 59 in v0.3.1; +2 for new cases).

### integration / remote-relay test script

- `tools/test_remote_relay.py` — **new**. 31 payload cases covering the quoting scenarios the base64 ssh trampoline is supposed to protect:
  - single/double/backtick nesting, mixed Python literals, `\n` in Python source
  - `$VAR` expansion vs single-quote literal, `A=42` inline assignments
  - pipes, pipe chains, `tr`, redirects, subshells (`(cd && pwd)`)
  - command substitution `$(...)`, backtick literals
  - multi-line with real LF, heredocs (`<<EOF` and `<<'EOF'`)
  - control flow (`&&` / `||` / `if` / `for`)
  - UTF-8 (Chinese, emoji, Japanese)
  - literal backslashes, no-match globs, ~1.5KB long payloads, nested Python JSON
- Modes: `LOCAL` (fake-ssh shim, default) and `--host HOST` (real remote). Prereq: running executor.
- Command: `python3 tools/test_remote_relay.py [--host H20] [--only <pattern>] [--stop-on-fail]`.

## Coverage

- `python3 -m unittest discover tests`: **61 passing, 0 failing**.
- `tools/test_remote_relay.py` against the local `tests/availability/ssh` shim, with an executor running in the same sandbox:

  **Context**: this sandbox has been continuously exercising ntfy.sh for days, so rate-limit effects are in force. The run hit HTTP 429 / `[Errno 99] Cannot assign requested address` (client-side socket exhaustion from retry storms) on ~2/3 of cases. The **cases that reached the remote side in time** all passed — including:
  - `basic_echo`, `uname`, `exit_nonzero` (basic relay)
  - `dquote_inside`, `squote_in_dquote`, `python_print_escape`, `python_tuple_dict`, `mixed_quotes_nested`-adjacent (quote nesting)
  - `backtick_literal`, `var_mix`, `pipe_chain`, `or_chain`, `for_loop`
  
  The failing cases **all** failed with `publish rejected … HTTP Error 429` or socket-exhaustion-during-retry, not with content mismatch. The base64 transport itself is sound — this is the same ntfy.sh rate-limiting situation documented in v0.2.1 and v0.3.

  To reproduce without rate-limit pressure: run against a fresh ntfy topic (change `Settings.ntfy_forward_topic` / `_backward_topic` temporarily) or a private ntfy instance.

## Live verification (v0.3.2 shell=False chain, manual trace)

**submit_gitbash.py 'ls'**:
```
envelope.command = "ls"
executor: Popen(["/bin/bash", "-c", "ls"], shell=False)
→ bash.exe parses "ls", runs ls, returns stdout
```
No cmd.exe anywhere. ✓

**submit_gitbash_ssh.py H20 'python3 -c "print(\"hi\")"'** (base64 path):
```
b64 = cHl0aG9uMyAtYyAicHJpbnQoXCJoaVwiKSIK  (example)
envelope.command = ssh H20 "bash -c \"$(echo 'cHl0...' | base64 -d)\""
executor: Popen(["/bin/bash", "-c", envelope.command], shell=False)
→ bash parses: ssh H20 "bash -c \"$(echo '...' | base64 -d)\""
→ ssh sends argv[2] to remote
→ remote $SHELL -c "bash -c \"$(echo '...' | base64 -d)\""
→ $(echo | base64 -d) → decoded = 'python3 -c "print(\"hi\")"'
→ remote inner bash -c 'python3 -c "print(\"hi\")"'
→ python3 prints 'hi'
```
Zero parse layers consume the payload until the final remote bash. ✓

## Residual risk (deferred to v0.4)

- Remote `base64` missing → silent success via empty `$()`.
- `$()` strips trailing newlines from decoded payload.
- ARG_MAX pre-flight.
- `submit_powershell_ssh.py` still uses non-base64 `--% wrapped_target`.
- Host option injection via leading `-`.
- `AET_SHOW_WIRE=1` debug env.

These are tracked in PLAN.md §10.
