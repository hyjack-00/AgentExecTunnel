# Evaluation v0.4

## Changes

### `submitter/submit_files.py` — full rewrite (~180 lines)

- Module docstring describes the flow + the four exit codes (0 / 1 / 2 / 3).
- New constants: `UPLOAD_RETRY_ATTEMPTS=3`, `UPLOAD_RETRY_DELAY_SECONDS=15.0`, `REMOTE_VERIFY_TIMEOUT_SECONDS=120`; `EXIT_LOCAL_UPLOAD_FAILED=1`, `EXIT_REMOTE_PULL_FAILED=2`, `EXIT_FILE_MISSING=3`.
- New helpers:
  - `_render_remote_verify_command(namespace, filename)` — emits a single-string bash command that pulls with 3×15 s retry and runs `[ -e ... ]`, tagging stdout `VERIFY_OK` or stderr `VERIFY_MISSING`, with distinct exit codes 10/11/12.
  - `_manual_retry_hint(verify_cmd)` — returns the `bash -c '<shlex.quoted>'` line the operator can paste on the executor host.
  - `_sync_forward_root(forward_root)` — best-effort `git_sync` (warning on failure).
  - `_push_with_retry(forward_root, message)` — 3 × 15 s outer retry, `git_commit_push(max_attempts=8)` inner.
  - `_run_remote_verify(namespace, verify_cmd)` — `publish_task(timeout_seconds=120)` + `wait_for_result`; returns `(exit_code, stdout_tail, stderr_tail)`; `exit_code is None` means stale or publish/wait error.
- New `main()` flow: sync → namespace-unique check → copy → push-with-retry → render → dispatch → block → interpret.

## Live verification

Executed the rendered remote verify command against three hand-built scenarios using a real `git init --bare origin.git` + `git init agent_forward` setup:

| Scenario | Expected | Observed |
|---|---|---|
| forward_root has the pulled file | `VERIFY_OK …` on stdout, exit 0 | ✅ `VERIFY_OK namespace=testns path=files/testns/mydir`, exit 0 |
| forward_root pulls clean but file absent | `VERIFY_MISSING …` on stderr, exit 12 | ✅ `VERIFY_MISSING namespace=missing_ns path=files/missing_ns/notfound`, exit 12 |
| `AET_FORWARD_ROOT` nonexistent | `forward_root not found …` on stderr, exit 10 | ✅ Matches. |

Not verified live: a network-failure scenario where `git fetch` keeps failing (exit 11). This requires a real unreachable remote; the retry structure is the standard `||` cascade and is tested for syntax by the happy path running (three `git fetch` attempts appear in the rendered command — confirmed by test assertion).

## Backward compatibility

- **Behavior change**: `submit_files.py` now **blocks**. Previously it returned as soon as the push completed (~5 s typical); now it waits for the executor to pull + verify (~10–60 s typical). Any automation that relied on fast return is affected.
- **Exit code change**: previously `submit_files.py` exited 0 on successful push, period. Now it exits 0 only on full end-to-end verification; exits 1/2/3 for the three specific failure modes. Scripts checking `$?` may need updating.
- **Namespace reuse change**: previously a re-run with the same `--name` would silently overwrite the staged path. Now it refuses with `SystemExit("…already in use…")`. Scripts that re-ran the same name to refresh content will break; the intended replacement workflow is "pick a new name" or delete the `files/<name>/` directory manually first.
- **Envelope format unchanged**: the verify task is an ordinary `kind=task` envelope with a shell `command` string. Nothing on the executor side is special-cased. A v0.3.x executor receives and runs the verify command correctly (it just sees a normal bash command). A v0.4 submitter paired with a pre-v0.4 executor still works — the executor pulls + verifies as expected, and the submitter interprets the result as usual.

## Residual risk

- See `reviews/v0.4.md` §"Residual risk". Summary:
  - `AET_FORWARD_ROOT` fallback assumes executor cwd is repo root; documented in README.
  - 120 s timeout not tunable via CLI yet.
  - Exit codes 10/11/12 are custom; an operator reading the remote stderr without consulting docs may be briefly confused.
  - No streaming progress during push or fetch.
  - No hash-based bit-integrity check — only presence.

## Verification targets for v0.5 / next cycle

- Real-remote round-trip with wall-time measurement.
- `--timeout-seconds` and/or `--max-retries` CLI flags.
- Consider extending the remote verify to optionally compute sha256 and compare against a submitter-side hash.
- Consider a `submit_files.py --list-pending` helper to show namespaces in flight or awaiting verify (if the concept of "pending" ever emerges).
