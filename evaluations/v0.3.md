# Evaluation v0.3

## Changes

### submitter (client-side rendering)

- `submitter/_submit_common.py::render_gitbash_ssh_command` — base64-wrap the payload, produce `ssh HOST "bash -c \"$(echo '<b64>' | base64 -d)\""` as the submitted command. Preview function unchanged (human-readable `ssh HOST '<payload>'`).
- `submitter/_submit_common.py::submit_and_wait` — signature `(label, command, timeout_seconds, metadata=None)`; removed `submit_mode` / `target_host`.
- `submitter/_submit_common.py` — removed `MODE_RELAY` / `MODE_SSH` constants.
- `submitter/submit_gitbash.py` — `submit_and_wait(label, payload, timeout)`.
- `submitter/submit_gitbash_ssh.py` — renders client-side, submits `relay_script` with `metadata={"ssh_host": host}`.
- `submitter/submit_powershell.py` — `submit_and_wait(label, payload, timeout)`.
- `submitter/submit_powershell_ssh.py` — renders via `render_ssh_command` client-side, submits the `powershell.exe -EncodedCommand …` string.

- `agent_exec_tunnel/executor.py::_execution_command` — deleted.
- `agent_exec_tunnel/executor.py::_run_task_worker` — calls `subprocess.Popen(task["command"], shell=True, ...)` directly.
- `agent_exec_tunnel/executor.py` — removed unused `import shlex`.
- `agent_exec_tunnel/config.py::PACKAGE_VERSION` — `v0.2.1 → v0.3`.
- `VERSION` — `v0.2.1 → v0.3`.

## Live verification

- Relay roundtrip: `AET_GIT_BASH_EXECUTABLE=/bin/bash python3 submitter/submit_gitbash.py --timeout-seconds 30 "echo v0.3-relay-…"` → `SUBMITTED command_id=…` + stdout replay + exit 0 within ~7s. Zero git push on the message plane.
- SSH base64 roundtrip: ntfy.sh anonymous tier was rate-limited (HTTP 429) during the E2E run, as documented in v0.2.1's evaluation. The submitter correctly surfaced `publish rejected; command was not published` with exit 1 (bounded publish semantics unchanged). The unit test `test_render_gitbash_ssh_command_uses_base64_wrap` exhaustively verifies the render output including base64 round-trip.
- Dead-code audit: `rg -rn "submit_mode|MODE_RELAY|MODE_SSH|_execution_command" --include="*.py"` in `agent_exec_tunnel/` and `submitter/` is **empty**. In `tests/`, remaining hits are intentional negative assertions or `ProbeSpec.submit_mode` (CLI-dispatch field, never touches envelope).

## Not run in this change

- Rate-limited environment prevented a fresh live ssh-through-shim E2E on ntfy.sh. Unit tests cover the render+decode path completely; for a fresh environment the shim smoke `PATH=tests/availability:$PATH` + submit_gitbash_ssh.py would confirm the last mile.
- Windows executor path (`submit_powershell*.py` via cmd.exe + powershell) untested in this sandbox. `render_ssh_command` output is mechanical and unchanged since v0.2; the unified-transport refactor only affects the call to `submit_and_wait`.
- `submit_files.py` concurrent-push sync regression — intentionally out of scope (listed as known issue in PLAN.md §9).

## Residual risk

- payload > ~100KB: base64 expansion × 4/3 + ARG_MAX ceiling on `sh -c` means very large payloads fail. Mitigation: ship as a file via `submit_files.py` and trigger with a short command.
- NUL bytes in payload as bash source: bash rejects regardless of transport — not a v0.3 regression.
- `$0` / `$BASH_SOURCE` lose file mapping in the decoded script.
- Remote `base64` binary required (essentially universal on Linux/macOS/BSD).

## Verification targets for next release

- Re-run the full live E2E when ntfy.sh rate-limits clear or on a private ntfy instance.
- Add an integration test that spawns a fake-ssh shim executor locally and exercises the full base64 round-trip for a `'"'`-heavy payload.
- Address the `submit_files.py` concurrent-push issue (probably by switching off git main-branch rebase-push to an object-store or per-file branch).
