# Evaluation v0.3.4

## Changes

### `submitter/_submit_common.py`

- Added `import re`.
- New module constants:
  - `_ARG_MAX_LIMIT = 100_000` — payload size ceiling (bytes).
  - `_HOST_PATTERN = re.compile(r"^[A-Za-z0-9._@:-]+$")` — ssh host charset.
- New helpers:
  - `_validate_host(host)` — rejects empty, leading `-`, and out-of-charset hosts.
  - `_check_arg_max(payload, label)` — enforces the 100 KB ceiling.
  - `_build_remote_trampoline(b64)` — emits the safety-checked remote shell script: `command -v base64` guard (exit 127), `[ -n "$_s" ]` guard (exit 97), `exec bash -c "$_s"`.
  - `_dquote_escape(value)` — escapes `\`, `` ` ``, `$`, `"` for bash `"..."` literals.
  - `_maybe_show_wire(command)` — prints `[wire] <full_command>` line iff `AET_SHOW_WIRE=1`.
- Rewrote `render_ssh_command` (PowerShell ssh path) to use the base64 trampoline under PowerShell single-quoting, replacing the old `--%` stop-parsing shape.
- Upgraded `render_gitbash_ssh_command` to use `_build_remote_trampoline` — same wire semantics as before, plus the two safety checks.
- All four renderers (`render_relay_command`, `render_ssh_command`, `render_gitbash_relay_command`, `render_gitbash_ssh_command`) now call `_check_arg_max` at entry; the two ssh variants also call `_validate_host`.
- All four `write_*_preview` functions now call `_maybe_show_wire` at the end.

### `tools/test_remote_relay.py`

- Preview stripper rewritten: instead of prefix-matching `-> ` / `  -> ` / `    -> ` / `SUBMITTED ` (which would eat a payload's output starting with arrow-like text), anchor on the `SUBMITTED command_id=` line. Everything before it (inclusive) is preview + meta; everything after is remote output. Robust to the new `[wire] ` line.

## Live verification

- **Local shim roundtrip**: With `PATH=$PWD/tests/availability:$PATH`, rendered a trampoline for `echo hello-world` via `render_gitbash_ssh_command('LOCAL', …)` and piped it through `bash -c "$(python3 -c ...)"`. Output: `hello-world`, exit 0. Confirms the generated script is syntactically valid bash AND the decode→exec path works end-to-end against a fake ssh target.
- **Missing-base64 guard**: Ran `bash -c 'export PATH=/tmp/empty; /bin/bash -c "command -v base64 >/dev/null 2>&1 || { echo missing >&2; exit 127; }"'` — prints `missing` to stderr, exits 127. Confirms the trampoline front-half behaves as designed when base64 is absent.

Real-remote verification against a ntfy-authed cluster is left to the operator; all submitter-side wire forms are deterministic from the unit tests.

## Backward compatibility

- **Submitter helpers that ONLY changed wire output shape**:
  - `render_gitbash_ssh_command` — wire script now has the safety prefix. An executor / remote shell that could run the v0.3.3 form can run the v0.3.4 form (strictly additive `command -v base64` check + `[ -n "$_s" ]` check; the happy path is unchanged).
  - `render_ssh_command` — wire shape changed (`--%` → PS single-quoted base64 trampoline). A Windows submitter that was calling this helper keeps working; an operator that scraped the wire for downstream automation (unlikely but possible) will see a different string.
- **Rejections new in v0.3.4**: A caller passing `host=""`, `host="-X"`, or a host with shell metachars now raises `ValueError` at render time instead of producing a broken ssh argv. This is a correctness fix (the old behavior silently produced something dangerous); anyone depending on those calls was already broken.
- **100 KB payload ceiling**: Previously silently forwarded, now rejected at submitter. If a caller was submitting multi-hundred-KB commands, they'll hit this immediately. Error message points them to `submit_files.py`.
- **Ntfy wire format unchanged**. v0.3.3 and v0.3.4 submitters / executors interop; the envelope bytes for any given payload are identical except when an ssh variant is used (where the `command` string embedded in the envelope is the trampoline script itself — already different between sub-versions in practice).

## Residual risk

- See `reviews/v0.3.4.md` §"Residual risk". Summary: permissive-but-not-dangerous host charset; `command -v` assumes POSIX-ish shell on remote; 100 KB ceiling is conservative for Linux, still a bit loose for Windows executors (but the right answer is `submit_files.py`); `AET_SHOW_WIRE` prints before publish is confirmed — all acceptable.
