# Evaluation v0.4.1

## Changes

### `agent_exec_tunnel/ntfy_transport.py`

- `publish_forever` grew a new kwarg `deadline_monotonic: float | None = None`.
- Two new early-exit paths (both return `False`):
  - Before attempting: `time.monotonic() >= deadline` → log `reason=deadline`
  - After an error, before sleeping: if `remaining <= 0` → log with `reason=deadline error=<exc>`
- Sleep is shortened to not overshoot the deadline.
- Backward-compat: `deadline_monotonic=None` preserves original infinite-retry semantics.

### `agent_exec_tunnel/executor.py`

- Imported `json`.
- New module-level constant `_TRUNCATION_NOTE`.
- New module-level helpers: `_envelope_size(envelope)`, `_truncate_result_envelope(envelope, budget_bytes)`.
- `_publish_result(result)` → `_publish_result(result, task_timeout_seconds)`. Truncates envelope via helper, derives deadline from task timeout, passes both to `publish_forever`.
- `_publish_ack(task_id)` → `_publish_ack(task_id, task_timeout_seconds)`. Same deadline treatment; no truncation (ACK envelopes are tiny).
- `_run_task_worker` reads `task["timeout_seconds"]` once and passes it to `_publish_ack`.
- `_finalize_result` reads `task.get("timeout_seconds")` with a fallback to `settings.default_timeout_seconds` (covers the `_publish_envelope_failure` path where the envelope itself was rejected and may not carry a valid timeout).

### `agent_exec_tunnel/config.py`

- `PACKAGE_VERSION`: `v0.4 → v0.4.1`.

## Live verification

Not run against a real VPN-audited relay in this sandbox (no such host
on hand). Unit tests exercise both the deadline path and the truncation
path comprehensively via mocked urlopen / mocked publish_forever.
Operator validation path:

```bash
# 1. Start executor with a tight budget to force truncation
AET_NTFY_RESULT_WIRE_BUDGET_BYTES=5000 python3 executor/run_executor.py

# 2. From another shell, submit a task that produces big stdout
python3 submitter/submit.py --timeout-seconds 15 \
  'python3 -c "print(\"X\"*50000)"'

# Expected: submitter receives status=done, stdout_tail begins with
# "[truncated by executor: original 50001B, envelope wire budget 5000B]"
```

```bash
# 1. Force publish failures by blocking ntfy egress (or breaking token)
sudo iptables -I OUTPUT -d 159.203.148.75 -j DROP

# 2. Submit a short-timeout task
python3 submitter/submit.py --timeout-seconds 10 'echo hi'

# Expected executor log:
# "ntfy publish_forever gave up topic=bwd attempts=N reason=deadline error=..."
# within ~10 s. Previously: spins forever.

sudo iptables -D OUTPUT -d 159.203.148.75 -j DROP
```

## Backward compatibility

- **Envelope wire format unchanged**. Truncation only shrinks existing
  fields (`stdout_tail`, `stderr_tail`); no new fields, no removals.
  Pre-v0.4.1 submitters receive truncated envelopes and display them
  normally — the `[truncated by executor: …]` prefix is just text.
- **`publish_forever` signature additive**. Callers that don't pass
  `deadline_monotonic` get identical behavior to v0.4.
- **`_publish_result` / `_publish_ack` signatures changed** — both
  gained a required `task_timeout_seconds` positional arg. Internal
  methods; no external API impact, but tests that mock them directly
  (none outside the test suite) would need updates.
- **New config field `ntfy_result_wire_budget_bytes`**. Defaults to 60
  KB which is safely under any realistic VPN audit cap. An operator
  with a stricter audit (e.g. 50 KB) should `export AET_NTFY_RESULT_WIRE_BUDGET_BYTES=40000`.

## Residual risk

See `reviews/v0.4.1.md` §"Residual risk". Summary: incoming poll
response aggregate still unprotected (mitigated by `ntfy_poll_since =
"10m"` but not bounded); `TailBuffer.limit` char-vs-byte unfixed;
truncation marker is in-band text, not a dedicated boolean field.

## Verification targets for v0.4.2 / next cycle

- Real-remote test against the SG-audited relay. Confirm: (a) a
  pathological 40 KB stdout_tail gets truncated and successfully
  publishes; (b) a wedged relay (everything audit-rejected) causes
  the executor to give up within task timeout rather than spinning.
- Decide whether to tighten budget default. 60 KB is conservative;
  some audited relays cap at 32 KB or 48 KB.
- Possible follow-ups: incremental poll (`since=<message-id>`) for
  the incoming side; `truncated: true` as a first-class envelope
  field.
