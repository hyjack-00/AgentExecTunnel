# Evaluation v0.3.3

## Changes

### `agent_exec_tunnel/ntfy_transport.py`

- New top-of-file constant: `NTFY_AUTH_TOKEN = ""` with a leading comment block explaining what to paste.
- New resolved value: `_NTFY_AUTH_TOKEN = os.environ.get("AET_NTFY_TOKEN", NTFY_AUTH_TOKEN)`. Env wins.

## Live verification

Not run against a real authenticated ntfy in this sandbox (no token on hand, and public ntfy.sh anonymous tier continues to rate-limit us). Unit tests cover the code path comprehensively via mocked `urlopen`. An operator with a ntfy Pro token can:

```python
# In agent_exec_tunnel/ntfy_transport.py
NTFY_AUTH_TOKEN = "tk_your_actual_token_here"
```

or equivalently:

```bash
export AET_NTFY_TOKEN="tk_your_actual_token_here"
python3 submitter/submit.py 'echo hi'
```

Both the submitter and the executor will automatically carry the `Authorization: Bearer …` header to every ntfy request.

## Backward compatibility

- Empty-token default (`NTFY_AUTH_TOKEN = ""`, `AET_NTFY_TOKEN` unset) produces byte-identical wire requests to v0.3.2 — no Authorization header, anonymous access. v0.3.2 and v0.3.3 processes interop with each other seamlessly when both run anonymous.
- When a token is set, all HTTP requests gain the header. If the submitter and executor have different tokens (or one has one and the other doesn't), they'll run against different authorization contexts — this is NOT interop, but NOT a v0.3.3 bug either.

## Residual risk

- Committing a token to git history if an operator edits `NTFY_AUTH_TOKEN` directly instead of using the env var. Mitigation: use `AET_NTFY_TOKEN` for production; the docstring at the top of the section recommends it.
- Bearer-only: hard-coded prefix `Bearer `. Basic auth would require an operator patch to `_auth_header()`. Tiny nit; ntfy docs prefer Bearer anyway.
- No token-rotation plumbing. Rotating the token requires restarting submitters and executors (so they re-read the env var). Acceptable for small deployments.
