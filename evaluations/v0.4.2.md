## Backward compatibility

- **Envelope wire format unchanged**. No new fields, no removals.
- **`Settings` adds one field** (`ntfy_poll_cap_seconds`). All other
  existing fields retain their types; default values changed for
  three poll knobs but the field names and semantics are stable.
  Operators pinning specific values via subclass / env override are
  unaffected.
- **Executor cap behavior changes**: if an operator was relying on
  `cap = default_timeout_seconds / 2` (e.g. had set
  `default_timeout_seconds=600` expecting a 300s poll cap), they
  now need to set `ntfy_poll_cap_seconds` explicitly. Not documented
  as a breaking change because the prior derivation was an
  implementation detail, not a contract.
- **`DEFAULT_FORWARD_REMOTE` changes from GitHub to Gitee**. Only
  affects fresh bootstraps. Existing `agent_forward` clones keep
  their configured remote. Operators outside China may want to
  `export AET_FORWARD_REMOTE=https://github.com/...` for speed.
- **No submitter API change**. `submit_task` / `publish_task` /
  `wait_for_result` / `submit_files.py` entrypoints unchanged.

## Residual risk

See `reviews/v0.4.2.md` §"Residual risk". Summary:

- v0.4.2 addresses volume + periodicity axes of the gateway's
  heuristic but does nothing for the client-diversity axis (still
  a single-user deployment).
- Gitee default may slow down out-of-region operators — easy
  override, not automatic detection.
- Submitter cadence unchanged; rarely a factor but noted.
