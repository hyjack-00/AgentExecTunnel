## Not run in this change

- Live E2E on Windows executor — this environment is Linux-only; the logic change is reviewed by hand-tracing (`cmd.exe /c "<git bash> -c <payload>"` → bash.exe starts → bash runs payload).
- Live E2E on Linux executor — ntfy.sh rate-limit still in effect from recent testing; unit tests cover submitter-side wiring.

## Residual risk

- Same as v0.3 (listed in reviews/v0.3.md and PLAN §8.2 "deferred"):
  - Missing remote `base64` → silent success via empty `$()`.
  - `$(...)` strips trailing newlines from decoded payload.
  - ARG_MAX cliff at ~95KB payload with no pre-flight.
  - PowerShell ssh path still not base64-protected.
  - Host option injection via leading `-`.
  - Doc drift `poll_since` 2h→30m.
- New: Linux users who need ssh on the tunnel don't have a dedicated submitter yet; they must hand-write the `ssh HOST "..."` string as the payload to `submit_bash.py`.

## Verification targets for next release

- Add `submit_bash_ssh.py` that reuses the shell-neutral base64 ssh trampoline (drop the git-bash outer wrap).
- ARG_MAX pre-flight in all render functions (raise `ValueError` with guidance).
- Trampoline safety: `set -e` or decoded-empty check so missing `base64` fails loud.
- `AET_SHOW_WIRE=1` env variable dumps the real envelope command after the preview lines.
- Port `render_ssh_command` to base64 trampoline (PowerShell ssh path).
- Shim-based integration tests that actually exec the rendered command through `/bin/sh -c`.
