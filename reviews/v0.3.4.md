# Review v0.3.4

## Scope

Hardening pass across `submitter/_submit_common.py`: close the concrete failure modes that had been parked in PLAN.md §9 as "v0.4 candidates". One patch per concern, all in the submitter layer — no changes to the executor, ntfy transport, or protocol. The pass also tightens `tools/test_remote_relay.py`'s preview-stripper, which had been a source of false PASSes when a payload's own output started with `-> `.

The theme is the same as v0.3.3: the codebase has matured enough that the remaining rough edges are specific, enumerable, and cheap to close individually — no re-architecture needed.

## Findings

### `submitter/_submit_common.py` — safety rail additions

1. **Host validation** (`_validate_host`). Before v0.3.4, `render_gitbash_ssh_command` and `render_ssh_command` accepted anything as `host` and dropped it into the ssh argv unchecked. A host like `-oProxyCommand=evil` was valid Python and valid shell, and ssh happily interpreted it as an option — a textbook ssh-option-injection vector that lets a malicious / buggy caller take arbitrary action on the submitter machine. Now the host must match `^[A-Za-z0-9._@:-]+$` AND must not start with `-`. Both renderers call the validator at entry. Rejection is a `ValueError` before any ntfy publish, so there is no wasted round-trip.

2. **ARG_MAX pre-flight** (`_check_arg_max` + `_ARG_MAX_LIMIT = 100_000`). Linux `MAX_ARG_STRLEN` is typically 128 KB per argv entry; previously a 200 KB payload would appear to submit fine, then fail opaquely at the executor with `E2BIG` from the `bash -c <command>` exec. Now all four render entry points (`render_relay_command`, `render_ssh_command`, `render_gitbash_relay_command`, `render_gitbash_ssh_command`) check and refuse with a message that points the user at `submitter/submit_files.py` for out-of-band content.

3. **Remote base64 trampoline safety** (`_build_remote_trampoline`). The old trampoline was `bash -c "$(echo '<b64>' | base64 -d)"`. Two silent-success bugs:
   - If the remote had no `base64` in PATH, `echo | base64 -d` would fail, `$()` would expand to empty, `bash -c ""` ran successfully, and the SSH call exited 0 — the user's command never executed, they never knew.
   - If b64 decoded to empty (garbled, truncated, locale glitch), same symptom.

   v0.3.4's trampoline:
   ```sh
   command -v base64 >/dev/null 2>&1 || { echo 'base64 not installed on remote' >&2; exit 127; }; \
   _s="$(echo '<b64>' | base64 -d)" && [ -n "$_s" ] && exec bash -c "$_s" || \
   { echo 'base64 decode failed' >&2; exit 97; }
   ```
   Three improvements: explicit `command -v` check with exit 127; non-empty check with exit 97; `exec` so the decoded command's exit propagates without a wrapper layer. Exit codes chosen to be distinct from common shell exits (126 "can't execute", 127 "not found" — we reuse 127 deliberately since that IS the semantic; 97 is outside the common ranges and stands out in logs).

4. **PowerShell ssh path also base64-ifies** (`render_ssh_command` rewrite). The prior shape used `ssh <host> --% <wrapped>` with PowerShell's `--%` stop-parsing token. That token is fragile: it stops PS parsing but does NOT stop remote shell parsing, so payloads with embedded double quotes or `$var` still chewed through the remote sshd's `$SHELL -c`. Now the same base64 trampoline as the git-bash path is used, wrapped in PowerShell single-quotes (which are literal, with `''` escaping a single quote — a no-op in practice because the trampoline doesn't emit single quotes). Result: payload-bytes-are-opaque parity between the two ssh-flavored submitters.

5. **`AET_SHOW_WIRE=1` debug switch** (in `_maybe_show_wire`, called from all four `write_*_preview` functions). When set, each submitter emits an extra `[wire] <full_command>` line after the three human-readable preview lines. The motivation: preview lines are intentionally human-legible ("`ssh HOST '<payload>'`"), but the actual bytes on the wire go through base64 + -EncodedCommand + cmdline wrapping, so they diverge from the preview. Previously an operator had to scrape ntfy to see what was actually sent. Now `AET_SHOW_WIRE=1 python3 submitter/submit_gitbash_ssh.py H20 'cmd'` shows both forms on stdout.

### `tools/test_remote_relay.py` — stripper hardening

The old stripper used prefix matching (`-> `, `  -> `, `    -> `, `SUBMITTED `). If a test payload emitted its own output starting with `-> ` (plausible for programs printing arrows or prompts), the stripper would swallow it as preview and the expected-output assertion would fail mysteriously. New stripper anchors on the `SUBMITTED command_id=` line: everything before that line (inclusive) is dropped, everything after is remote output. This is also robust to the new `[wire] ` line from `AET_SHOW_WIRE=1`.

## Deliberate non-choices

- **No basic-auth variant for ntfy**. v0.3.3 left this as a candidate; nobody has shown up with a basic-auth ntfy server, and the user can hand-set `NTFY_AUTH_TOKEN = "Basic <b64>"` if they really need to (with a one-line patch to `_auth_header`). Not worth a config knob yet.
- **No token redaction in logs**. v0.3.3's rationale still holds — ntfy 401/403 responses don't echo the token, and the retry log includes the HTTP error repr which is body-less for auth failures.
- **No per-task shell hint in envelope**. Kept out deliberately: v0.3's unification guarantee (executor is mode-agnostic, just runs the string) is more valuable than mixed-shell convenience on a single executor. Operators can run two executors on different ntfy topics if they truly need both bash and powershell workloads.
- **ARG_MAX limit at 100 KB, not the actual system limit**. Keeps the submitter error deterministic regardless of kernel config / PAGESIZE. An operator running on a beefy Linux where `MAX_ARG_STRLEN` is higher can relax this by patching the constant.
- **No backward-compatibility shim for the old `--%` form**. Nobody outside this repo was calling `render_ssh_command` directly; it's an internal submitter helper. Cleanly rewriting it is better than leaving dead flags.

## Residual risk

- **Host charset still permissive**. `user@host:port` works, but so does `user@host:0;weird` — wait, `;` is NOT in the charset, `;weird` is rejected. Re-verified: the charset is strict enough. The only "risk" is legitimate hostnames that use characters outside `[A-Za-z0-9._@:-]` (unicode hostnames — exceedingly rare in practice; operators with these can patch the regex).
- **`command -v base64` is itself a shell builtin check**. If the remote shell is not POSIX-compatible (e.g., some stripped busybox shells), `command -v` may not be available. Unlikely in any real ssh target; we accept it.
- **100 KB ARG_MAX limit is a soft shield**. A payload of 99 KB still travels fine on Linux but may exceed Windows `CreateProcess` argv (32 KB) when routed through cmd.exe. Since v0.3.2's shell=False removes the cmd.exe hop, this matters less, but it can still bite a submitter running on Windows against a Windows executor. Not addressed here — the right fix is submit_files.py anyway.
- **`[wire]` line is printed even if publish fails**. The preview writer runs before `submit_and_wait`'s publish; if publish fails, the wire line was still printed. Acceptable for a debug switch.
