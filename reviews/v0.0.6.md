# Review v0.0.6

## Scope

- restore legacy-style long-running executor behavior for transient git/network failures
- add executor-side infinite retry/backoff around sync and durable push paths; submitter remains bounded-retry
- convert task subprocess timeout into a durable `stale` result
- document the continuously-running executor model in design/docs

## Findings

- No blocking findings remain for `v0.0.6`.
- The prior `v0.0.5` executor still had one serious operational gap: transient git failures and task timeout could terminate the long-running process.
- The current executor now matches the intended deployment rule more closely:
  - separate submitter/executor clones against the same remotes
  - executor keeps retrying sync/push until connectivity returns
  - task timeout is written into backward as one final stale result
- The unsafe shared-worktree case is still unsupported; this change hardens the supported separate-clone model rather than changing that boundary.

## Residual Risk

- Submitter still uses bounded local waiting; caller timeout remains a local outcome rather than a remote state mutation.
- ACK-only tasks can still exist if a process is interrupted after ACK becomes durable but before final result is durably pushed.

## Decision

Release `v0.0.6`.
