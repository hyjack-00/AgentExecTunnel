# Review v0.1.2

## Scope

- bootstrap / burst / executor ergonomics fixes carried over from the v0.1.1 review (items E–I)
- scan-log rewrite around the task state model
- `DESIGN.md` polish: explicit task state machine, reconciliation rules, diagram pruning

## Findings

### Ergonomics (E–I)

- `tools/run_burst_live.py` and `tools/run_burst_local_relay.py` now propagate `branch` out of `load_remote_urls()` into `attach_repo_clones` / `sync_repo_to_remote_url`. Non-`main` branches set via `AET_DATA_BRANCH` are no longer silently dropped.
- `executor/run_executor.py` runs a preflight check that raises `SystemExit` with a `run python3 tools/bootstrap_repos.py ... first` hint if either data repo is missing `.git`. The prior behavior was to descend into an opaque fetch/clone retry loop.
- `tools/bootstrap_repos.py` now catches `CalledProcessError` around `ensure_repo` and re-raises `SystemExit("git command failed: <cmd>\n<stderr>")`. Auth / network / missing-branch failures surface as a one-line readable error rather than a traceback.
- `tests/test_cli_entrypoints.py::test_bootstrap_ensure_repo_rejects_non_empty_non_git_dir` locks in the safety check that `ensure_repo` refuses to clone into a populated non-git directory.
- `README.md` gains a "Migrating from the old submodule layout" section documenting the `submodule deinit` / `rm -rf .git/modules/...` cleanup path.

### Scan log

- The old `scan pending=N` line conflated `scanned` (which includes pre-ACK'd tasks) with truly pending work, making it look like new tasks were being ignored when in fact they were all already in `running` state from a prior executor run.
- Renamed to the state-model vocabulary (`claimed` / `running`) and made the line suppress zero-valued fields, so an idle executor stays silent and the output of a busy scan reads as counts-per-state.
- Dropped `finished=` — in the 2-hour scan window terminal tasks dominate and the count carries no actionable signal.

### Design doc

- `Task State Model` now names three observable states (`unclaimed`, `running`, terminal) and includes a `stateDiagram-v2` plus a `Main-Loop Reconciliation` subsection describing exactly what the loop does per state, including orphan-`running` reconciliation by ACK-age.
- `Interrupted Finalize` no longer says "not auto-reclaimed"; it defers to the reconciliation rule. This is a design statement — the implementation still freezes orphan-`running` into `blocked_tasks`; see Residual Risk.
- The `Why Shared Worktree Is Unsafe` sequence diagram is replaced with prose. The remaining two mermaid diagrams (end-to-end flow, dispatcher/worker/writer) address separate concerns and stay.

## Residual Risk

- Design now prescribes main-loop reconciliation of orphan-`running` tasks, but the executor still imports ACKs into `blocked_tasks` at startup and never revisits them. This is called out in `PROGRESS.md` as two follow-up items (implementation + state-edge tests) and is the largest gap between design and runtime in this release.
- The `tests/test_submitter_flow.py::test_multiple_submitters_can_publish_without_forward_window_limit` case is order-sensitive under `unittest discover` — it passes in isolation but errors when another integration test leaves cwd or filesystem state behind. Not introduced by this release; logged for cleanup.

## Decision

Release `v0.1.2`.
