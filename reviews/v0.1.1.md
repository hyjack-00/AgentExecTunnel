# Review v0.1.1

## Scope

- drop git submodules for `agent_forward` / `agent_backward`
- introduce `agent_exec_tunnel/remotes.py` as the single source of data-repo URLs (env / file / default)
- rewrite `tools/bootstrap_repos.py` as a clone-or-sync helper that works without submodule machinery
- update README startup flow and burst/test tooling to match

## Findings

- Submodule pinning was structurally wrong here: every submit/ack/result produces a new commit in the data repos, so the tunnel-recorded submodule SHA drifts constantly. A fresh clone on another machine would fail with `not our ref <sha>` whenever a local runtime commit had not been pushed to the declared remote. A user hit exactly this symptom during startup of the current checkout.
- Moving the data repos to gitignored sibling clones provisioned by bootstrap removes the entire ghost-SHA failure class. The tunnel commit no longer tries to pin something that cannot be meaningfully pinned.
- `agent_exec_tunnel/remotes.py` resolves URLs from `AET_FORWARD_REMOTE` / `AET_BACKWARD_REMOTE` / `AET_DATA_BRANCH`, then an optional `.aet-remotes.json` at the tunnel root, then the public GitHub defaults. CLI flags on `bootstrap_repos.py` override all three.
- `bootstrap_repos.ensure_repo` is idempotent: first call clones, subsequent calls `fetch + checkout -B main + reset --hard origin/main`, and it rewrites `remote.origin.url` if the caller supplies a new URL.
- `tests/test_fresh_clone.py` now injects file-based bare remotes through env vars rather than faking submodule URL rewrites.

## Residual Risk

- Existing machines that still have `agent_forward/` and `agent_backward/` as submodule working trees (with `.git` as a gitdir-redirect file) will keep working through the redirect, but are in a legacy shape. Next rebuild should re-clone cleanly.
- Users on machines with stale `.git/modules/agent_forward` / `.git/modules/agent_backward` under the tunnel repo can safely delete those after moving off submodules, but bootstrap does not prune them automatically.
- `tools/bootstrap_repos.py` no longer supports the old `var/local_remotes/` auto-localization path; local-bare-remote workflows now go through explicit `--forward-url` / `--backward-url` or env overrides.

## Decision

Release `v0.1.1`.
