# Review v0.3.2

## Scope

User re-clarified the requirements:
1. Quote nesting (`'` / `"` / `` ` `` arbitrary depth) must **always** be solvable (base64 allowed).
2. Structure must stay simple — ssh commands should look like `"C:\...\bash.exe" -c "ssh HOST '<target>'"`, **no secondary wrapping on Windows**.
3. `submit_*_ssh.py` variants are purely convenience. `submit_gitbash_ssh.py H20 'cmd'` must be user-observable equivalent to `submit_gitbash.py 'ssh H20 \'cmd\''`.
4. EXTRA:
   - main: executor must be maximally adaptable — no payload theoretically unreachable due to missing transformation layer. Independent of string vs argv transport.
   - secondary: prefer a single nested payload string over code-defined argv lists; a `submit.py` CLI should be able to express any variant's effect via a hand-crafted payload.

The v0.3 → v0.3.1 path over-engineered requirement 3: it wrapped the payload in a Windows cmdline at the submitter so `cmd.exe /c` would then start bash.exe. That worked, but it added one layer the user explicitly said "不要". v0.3.2 fixes this by changing the executor.

## Findings

### Executor side — `shell=False` with configured shell

- `agent_exec_tunnel/config.py` — new `Settings.executor_shell` and `Settings.executor_shell_args`. Defaults pick `/bin/bash` on POSIX, prefer Git Bash (`C:\Program Files\Git\bin\bash.exe` or the x86 variant) on Windows, fall back to `cmd.exe`. Env overrides `AET_EXECUTOR_SHELL` / `AET_EXECUTOR_SHELL_ARGS`.
- `agent_exec_tunnel/executor.py` — the worker's `subprocess.Popen` switches from `shell=True` + command string to `shell=False` + `[shell, *args, command]`. This bypasses Python's hardcoded `/bin/sh` / `cmd.exe` wrap.
- **Effect**: on Windows, `cmd.exe /c` is no longer in the critical path. The envelope's `command` string is fed **directly** as the `-c` argument of the configured shell.

### Submitter side — envelope stays a plain string

- `submitter/_submit_common.py` — unchanged on the wire: `submit_and_wait(label, command: str, timeout_seconds, metadata=None)`. Still string, not argv list.
- `submitter/submit_gitbash.py` — v0.3.1 wrapped the payload client-side in `"<git bash>" -c …` because the executor's old `cmd.exe /c` needed it. v0.3.2 removes that wrap — submit the raw payload and let the executor's bash do the parsing. Preview (`write_gitbash_relay_preview`) is unchanged: still shows `-> "<git bash>" -c <payload>` as the human intent.
- `submitter/submit_gitbash_ssh.py` — similarly reverts v0.3.1's outer-cmdline wrap. Now submits `relay_script` (the ssh+base64 trampoline string) directly. The executor's bash parses the script, invokes ssh, remote bash decodes the base64 payload and runs it.
- `submitter/submit_bash.py` — unchanged from v0.3.1 (raw payload).
- `submitter/submit_powershell.py` — unchanged from v0.3.1 (builds `powershell.exe -EncodedCommand <b64>` string). The executor's bash invokes powershell as a subprocess — no cmd.exe needed here either.
- `submitter/submit_powershell_ssh.py` — unchanged from v0.3.1.

### New: `submitter/submit.py`

The bottom-of-stack CLI. The envelope's `command` is **exactly** what the user types — no encoding, no wrapping, no helper. Every other `submit_*.py` flavor is reproducible by hand via this CLI:

```bash
# submit_gitbash.py 'ls /tmp'               ⇔
python3 submitter/submit.py 'ls /tmp'

# submit_gitbash_ssh.py H20 'cmd'           ⇔
python3 submitter/submit.py 'ssh H20 "bash -c \"$(echo '\''<b64>'\'' | base64 -d)\""'

# submit_powershell.py 'Get-Location'       ⇔
python3 submitter/submit.py 'powershell.exe -EncodedCommand <b64>'
```

This satisfies EXTRA.secondary: the helpers are opt-in simplification over an always-available raw path.

### New: `tools/test_remote_relay.py`

Standalone end-to-end test script. Runs 30 payload shapes through `submit_gitbash_ssh.py` and verifies each round-trips intact:

- quote nesting — `\"`, `\'`, mixed, escaped Python string literals
- `$VAR` expansion vs literal (single-quote on remote)
- pipes, command substitution `$(...)`, backticks `` `...` ``
- redirects, subshells `(cd && pwd)`, heredocs (with and without variable expansion)
- control flow (`&&` / `||` / `if`/`for`)
- real-newline multi-line payloads
- UTF-8 (Chinese, emoji, Japanese)
- literal backslashes, glob-no-match, ~1.5KB long payloads
- nested Python-in-shell (`python3 -c '…'`)

Modes:
- default: uses `tests/availability/ssh` fake-ssh shim → local round-trip, no network
- `--host HOST`: real SSH target. Same script, different target.

Prereq: an executor must already be running; the script is stateless and does not manage executor lifecycle.

## Requirement check

| User requirement | v0.3.2 status |

| (1) Quote nesting always solvable | ✅ relay: 0 parses at transport + bash -c 1 parse. ssh: base64 trampoline → 1 parse at final remote bash -c |

## Residual risk (still deferred to v0.4)

- Remote `base64` missing → `bash -c "$(empty)"` exits 0 silent success.
- `$(...)` strips trailing newlines from decoded payload (rarely observable, still real).
- ARG_MAX preflight — payloads > ~95KB fail late.
- `submit_powershell_ssh.py` quoting is still the `--% wrapped_target` form, not base64.
- Host option injection via leading `-` in hostname.
- `tests/availability/` descriptive drift (mentions CLIs by executor OS which no longer applies).
- `AET_SHOW_WIRE=1` debug env for dumping the real envelope command next to the preview.

These are tracked in PLAN.md §10 as the v0.4 bundle.
