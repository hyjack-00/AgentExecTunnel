# Review v0.2

## Scope

- Replace the GitHub-based message plane (task / ACK / result JSON in two git repos) with ntfy.sh pub/sub on two fixed topics `agent-forward-285` and `agent-backward-285`. File transfer (`submitter/submit_files.py` → `agent_forward/files/<ns>/…`) intentionally stays on GitHub.
- Dead-code cleanup of the entire ACK / orphan reconciliation / `GitWriter` / `backward_root` surface.
- Restore the legacy `agent_forward` v4.5 availability dashboard on top of the ntfy-transport branch.
- Address the adversarial subagent review against this branch before tagging.

## Findings

### Transport

- `agent_exec_tunnel/ntfy_transport.py` holds a stdlib-only (`urllib.request`) implementation of `publish` / `publish_forever` / `poll_since` / `poll_loop` / `wait_for` / `seed_seen_ids`. No new runtime dependencies.
- Forward-publish is bounded: `publish()` retries up to `publish_max_attempts` with exponential backoff and honors `Retry-After`. Submitter raises `NtfyPublishError` to the caller on final failure — "command was not published" is a first-class visible state.
- Backward-publish is infinite: `publish_forever()` retries until success (or until the executor is stopped), exponential backoff capped at 30s, honors `Retry-After`. Each attempt builds a fresh `urllib.request.Request` with `Connection: close` so no pooled TCP/TLS session can rot across a long outage.
- Poll cadence is 1s base with upward jitter that grows on both idle **and** error, capped at `timeout/2` (default 300s → 150s). With `growth=1.10`, cap is reached in roughly 50 empty polls. Any new envelope resets the jitter to 0.
- `poll_loop` consumes a `is_seen(task_id) -> bool` callback instead of a raw set; the caller owns the lock around its dedup state. Fixes the CRITICAL cross-thread `seen_ids` data race the review flagged.

### Executor

- `agent_exec_tunnel/executor.py` shrinks from 588 LOC to ~260 LOC. Removed: `GitWriter` class + `WriteRequest` + writer thread + `_ensure_repo`, `startup_scan*` / `scan_recent*` / `_git_sync_once` / `_bucket_glob`, `_recover_from_backward`, `_reconcile_orphan_stale`, `orphan_ack_at` / `claiming_tasks` / `blocked_tasks` fields.
- Main thread polls forward topic; each new envelope claims `task_id` into `running_tasks` (under `_state_lock`), validates `timeout_seconds` + `command`, spawns a `threading.Thread`, returns. Worker thread runs `subprocess.Popen(shell=True, ...)`, polls the child, and calls `_finalize_result` → `_publish_result` → `publish_forever` itself. Only after publish success does it mark `seen_ids` and clear `running_tasks`.
- Consequence: a backward-topic outage blocks just the affected worker threads; the polling main thread is unaffected and can keep dispatching new tasks to new workers.
- Executor startup primes `seen_ids` via a one-time `seed_seen_ids(backward_topic)` — a restart within the 2h replay window does not re-run tasks that already have a result envelope.
- Task-envelope validation: missing / non-positive `timeout_seconds` or missing `command` publishes a `failed` result (via the same `_finalize_result` state machine) with an explanatory `stderr_tail`. No more dispatching ill-formed envelopes straight into `subprocess.Popen`.

### Submitter

- `publish_task()` does a single ntfy POST; if `publish()` fails, the caller sees `NtfyPublishError` immediately. Removed the outer 3-round retry that multiplied into 9 POSTs per submit.
- `wait_for_result()` polls the backward topic via `ntfy_transport.wait_for`, cap = `timeout/2`. Deadline = `timeout + submit_timeout_grace_seconds` (default 15s). The grace absorbs publish → dispatch → worker-spawn skew so the submitter can observe an executor-authored `stale` envelope instead of racing past it.
- `new_task_id()` now appends `secrets.token_hex(8)` (64 bits) instead of 16 — within-second collision bound moves from ~300 ids to ~5×10^9 before birthday collision is likely. Fixes the MEDIUM review finding on burst collisions.
- `timeout_exit` differentiates "ntfy unreachable" vs "ntfy reachable, executor silent".

### Adversarial review follow-up

A dedicated subagent review against the branch raised 20+ findings. The CRITICAL / HIGH items were all addressed in-branch before tag:

- CRITICAL: `poll_loop` / `seen_ids` race — fixed by the `is_seen` callback indirection.
- HIGH: silent result loss on backward publish failure — fixed by `publish_forever`. Chosen over the earlier "spool to pending_results then drain on next tick" approach because the user opted to trust network recovery and have workers block, rather than introduce a second retry tier.
- HIGH: submitter timeout racing executor's stale result — fixed by `submit_timeout_grace_seconds`.
- MEDIUM: doc drift (DESIGN.md / README.md / SKILL.md still described the old git transport) — rewritten.
- MEDIUM: publish retry multiplication (3×3=9) — collapsed to one layer.
- MEDIUM: `new_task_id` 16-bit jitter — bumped to 64 bits.
- MEDIUM: `preflight_check` insisting on `agent_forward/.git` — softened to a warning.
- MEDIUM: `build_report` / `render_html` duplicated rendering code — `build_report` deleted.

Deferred to post-v0.2 (documented in DESIGN.md "Known MVP trade-offs"):

- No ACK / at-most-once: mid-task crash + restart within 2h may re-run once.
- No auth on world-readable topics: anyone guessing `agent-forward-285` can inject a task; `subprocess.Popen(shell=True)` inherits the executor's full env + cwd + `~/.ssh`. Add HMAC signing or a private ntfy instance for production.
- No stdout/stderr streaming beyond 4KB tail: large-log tasks should write to files and upload via `submit_files.py`.
- Windows `shlex.quote` vs `cmd.exe` edge cases on the ssh-wrap path.

- End-to-end on real ntfy.sh: `submit_gitbash.py "echo ..."` round-trip completes in ~7s; zero git push on the message plane.

## Residual risk

- Worker thread blocks on `publish_forever` during a sustained backward-topic outage. `seen_ids` / `running_tasks` stay consistent, but the task count in `running_tasks` grows during the outage (every newly finalized task parks on its own thread). Memory pressure is bounded by thread count × 4KB tail buffers; for realistic workloads this is negligible. If the executor is forcibly killed during the outage, the affected tasks lose their result envelope — documented as the same MVP trade-off as mid-task crash (one re-run on restart within 2h).
- `publish_forever`'s exponential backoff caps at 30s. An executor stuck publishing into a persistently dead ntfy will not escalate further — no alert. Operators should watch the executor log; `publish_forever attempt=N ... retry_in=30.0s` is the signal.
- The main poll thread's `is_seen` check acquires `_state_lock` briefly per envelope. Under high-burst traffic (thousands of envelopes per poll snapshot) this is still fast (microseconds), but if it ever becomes a problem the callback can switch to a lock-free snapshot.
