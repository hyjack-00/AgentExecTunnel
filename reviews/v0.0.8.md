# Review v0.0.8

## Scope

- reintroduce legacy-style active-process monitoring for all tasks
- restore legacy-style submitter CLI output and timeout wording
- keep the dual-repo protocol but align executor/runtime logging more closely with `agent_forward`

## Findings

- No blocking findings remain for `v0.0.8`.
- The biggest runtime mismatch from `agent_forward` was not git retry anymore, but task lifecycle shape:
  - current repo had become one-task blocking after ACK
  - legacy behavior kept active tasks in memory and re-checked them from later scan passes
- This version restores that active-set model for all tasks, not just ssh-flavored tasks.
- User-facing submitter behavior now also matches legacy much more closely:
  - preview first
  - `SUBMITTED command_id=...`
  - timeout wording
  - stdout/stderr tail replay
- Protocol storage remains dual-repo and still uses `task_id` internally; `command_id` is now a user-facing alias in CLI/log output.

## Residual Risk

- `ack only` is still possible if the process is interrupted after ACK becomes durable but before any final/stale result becomes durable.
- `stale` intentionally allows the local process to outlive the protocol deadline; cleanup is local-only after that point.

## Decision

Release `v0.0.8`.
