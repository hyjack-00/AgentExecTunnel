# Review v0.3.3

## Scope

Small feature add: a hardcoded `NTFY_AUTH_TOKEN` constant at the top of `agent_exec_tunnel/ntfy_transport.py`. When non-empty, every outbound HTTP request (publish, poll, attachment fetch) carries `Authorization: Bearer <token>`. When empty (default), behavior is identical to v0.3.2 (anonymous ntfy.sh).

The motivation is operational: the public `ntfy.sh` anonymous tier has rate limits (HTTP 429) and per-client port ceilings that repeatedly bit us during v0.2.1/v0.3/v0.3.2 live tests. With a ntfy Pro token or a private ntfy instance behind a reverse proxy, the same `Bearer` auth unblocks both submitter and executor in one edit.

## Findings

### `agent_exec_tunnel/ntfy_transport.py`

- **New module-level constant** at the top of the file: `NTFY_AUTH_TOKEN = ""`. Commented so an operator knows exactly what to paste.
- **`_NTFY_AUTH_TOKEN` resolution**: `os.environ.get("AET_NTFY_TOKEN", NTFY_AUTH_TOKEN)`. Env wins over the hardcoded default. This lets an operator leave the repo clean and put the token in a systemd unit, docker env, or `.env` file consumed by the process supervisor.
- **`_auth_header()` helper**: returns `{"Authorization": f"Bearer {_NTFY_AUTH_TOKEN}"}` when set, else `{}`. Cheap; called once per request.
- **Three injection points** (all covered):
  - `_publish_once` — used by `publish()` (bounded) and `publish_forever()` (infinite). Merges the auth header into the existing `Content-Type` / `Connection: close` map.
  - `poll_since` — used by `poll_loop`, `wait_for`, and `seed_seen_ids`. Was previously a pure GET with no headers; now carries only the auth header (other defaults are fine).
  - `_load_json_url` — the attachment fallback for oversize messages. Private ntfy sets up attachments behind the same auth as the main endpoint, so this request needs the header too.

### Unit tests (`tests/test_ntfy_transport.py`)

Five new cases under `AuthHeaderTests`:
- `_auth_header` with empty token → `{}`.
- `_auth_header` with `"tk_example12345"` → `{"Authorization": "Bearer tk_example12345"}`.
- `_publish_once` with a mocked `urlopen` and a non-empty token → the Request object carries `Authorization: Bearer tk_abc`.
- `_publish_once` with empty token → Request has no Authorization header.
- `poll_since` with a mocked `urlopen` and a non-empty token → Request carries the header.

The token is injected via `mock.patch.object(ntfy_transport, "_NTFY_AUTH_TOKEN", <value>)` so tests don't touch the environment.

Full suite: **66 passing**, 0 failing (was 61 in v0.3.2; +5 auth tests).

## Deliberate non-choices

- **No Basic auth support**. Ntfy supports both `Authorization: Basic <b64(user:pass)>` and `Authorization: Bearer <token>`. Bearer is their recommended modern pattern (access tokens with fine-grained topic ACLs); Basic is mostly for legacy. User can still do Basic by hand-setting the constant to `"Basic <b64>"` — `_auth_header()` just prefixes "Bearer " — **actually no, current code hardcodes `Bearer `**. If a user needs Basic, they'd need to patch the helper. Noted as a tiny v0.4 candidate.
- **No token redaction in logs**. `publish_forever`'s retry log includes the HTTP error repr, which for a 4xx would include the body — ntfy's 401/403 responses do not echo the token. Safe enough.
- **No Settings-level plumbing**. Kept as a module-level constant + env var. Cleaner than threading through `Settings` / `NtfyConfig`, and matches the user's "自己来填" intent.

## Verification

- `python3 -m unittest discover tests`: **66 passing**.
- Manual check: `_auth_header()` at import time with `NTFY_AUTH_TOKEN = ""` → `{}`. Set env `AET_NTFY_TOKEN=tk_test123` before import → header is `{"Authorization": "Bearer tk_test123"}`.
- Backward compatibility: empty-token (default) code path is byte-identical to v0.3.2 on the wire. v0.3.2 clients/executors and v0.3.3 clients/executors interoperate freely as long as everyone runs with the same token (or all run anonymous).

## Residual risk

- If you paste a token into `NTFY_AUTH_TOKEN` and `git add` the file, the token lands in commit history. Use the `AET_NTFY_TOKEN` env var for production.
- Private ntfy instances typically require HTTPS + valid TLS certs. `truststore.inject_into_ssl()` (soft import in `ntfy_transport.py`) covers host-CA chains; if your ntfy is behind a corporate CA, install `truststore` on the executor host.
- Token auth alone is not sufficient on a **shared** ntfy instance where the token authenticates the account but not the topic. Topic-level ACLs (ntfy Pro) or a dedicated private ntfy server are needed for adversarial multi-tenant safety.
