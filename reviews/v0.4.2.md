# Review v0.4.2

## Scope

Two small, operationally-driven adjustments based on a real incident:
ntfy.sh went dark for several hours at a time on a Huawei corporate
network (proxy `proxyhk.huawei.com:8080`, `netentsec`-family gateway),
then auto-recovered on the same order of magnitude. Multi-round
diagnostic pushed out the first two theories (SNI blacklist; corporate
proxy destination blocklist) — the proxy reliably returns
`HTTP/1.1 200 Connection established` on `CONNECT ntfy.sh:443`, and
initial `curl 000`s were SChannel revocation-check errors
(`0x80092012`) unrelated to the block. The multi-hour TTL plus
auto-recovery excludes short-time rate limiting too. The surviving
hypothesis is the gateway's **new-domain auto-isolation** policy:
novel destinations that attract persistent periodic traffic get
tossed into an automatic isolation bucket for a few hours before
auto-clearing.

Polling alone cannot defeat that class of heuristic (volume +
periodicity + client-diversity are each independent triggers, and
this deployment is a single-user source), but bringing daily GET
volume down and widening the interval band from `[3, 150]s` to
`[5, 300]s` lowers the trigger surface on one axis. In parallel,
the file-plane default remote was swapped from GitHub to Gitee —
Gitee is an order of magnitude faster in-country for push/pull and
is not treated as a "new domain" by the gateway. Message plane stays
on ntfy.sh for now pending observation data.

Non-goals for this release:

- **Not** migrating the message plane to Gitee. That's a possible
  v0.5 direction if ntfy.sh keeps getting isolated after v0.4.2
  observation — tracked in `PLAN.md` §9.
- **Not** switching to ntfy SSE / long-poll. A stable long-lived
  connection raises a different red flag (single-TCP beacon) that
  the same class of gateway hunts more aggressively. Deliberate
  non-change.
- **Not** introducing authentication, encryption, or envelope
  signing. Those are orthogonal and still tracked in the "Ntfy auth"
  MVP trade-off.

## Findings

### `agent_exec_tunnel/config.py`

- `PACKAGE_VERSION`: `v0.4.1 → v0.4.2`.
- `ntfy_poll_base_seconds`: `3.0 → 5.0`. Slower minimum cadence —
  reduces the absolute floor on periodic-signal energy. Submitter
  side adds ~2s worst-case initial poll delay; user accepted.
- `ntfy_poll_jitter_floor`: `0.05 → 1.0`. The per-miss additive term
  is now meaningful from the first miss rather than microscopic, so
  the ramp reaches meaningful jitter in a handful of misses instead
  of dozens. Matches the intent that jitter is a *time* quantity,
  not a precision knob.
- `ntfy_poll_jitter_growth`: unchanged at `1.5` (growth multiplier;
  geometric with floor gives an adequately fast ramp).
- New field `ntfy_poll_cap_seconds: float = 300.0`. Previously the
  executor derived `cap_seconds = default_timeout_seconds / 2 = 150s`,
  coupling idle-poll cadence to task-semantics (timeout). Those two
  things aren't the same concept: timeout is per-task, cadence is a
  broker-politeness + beacon-reduction knob. Separating them lets
  operators raise the cap to 600s without having to raise the task
  timeout ceiling along with it.

### `agent_exec_tunnel/executor.py`

- Line 194: `cap_seconds = self.settings.default_timeout_seconds / 2.0`
  → `cap_seconds = self.settings.ntfy_poll_cap_seconds`. No other
  behavior change. Submitter side (`wait_for_result` in
  `submitter.py:83`) still derives its own cap from
  `result_timeout_seconds / 2` — untouched, because submitter cadence
  is intrinsically tied to user-facing latency.

### `agent_exec_tunnel/remotes.py`

- `DEFAULT_FORWARD_REMOTE`:
  `https://github.com/hyjack-00/agent_forward.git`
  → `https://gitee.com/hyjack-00/agent_forward.git`.
- Env override (`AET_FORWARD_REMOTE`) unchanged — operators who
  prefer GitHub can still set it explicitly.
- Motivation: in the target deployment region (Chinese corp network)
  Gitee is 10–50× faster for `git fetch / reset --hard`, which
  shrinks the `submit_files.py` verify-loop wall time from
  "minutes sometimes" to "seconds reliably", and a domestic origin
  is much less likely to enter corporate new-domain isolation.
- The executor does **not** care which git host it pulls from — the
  `submit_files.py` verify command is rendered client-side; the
  executor just runs `git fetch + reset --hard` against whatever
  remote is configured on the bootstrapped `agent_forward` clone.
  Operators who already bootstrapped against GitHub keep working;
  new bootstraps get Gitee unless overridden.

### `PLAN.md`

- §9 "未解决" compacted: the hours-long diagnostic thread about
  SNI blacklist vs. proxy blocklist vs. CF rate-limit (none of which
  survived the evidence) is collapsed into a single line citing the
  surviving hypothesis (new-domain auto-isolation + TTL) and the
  escalation sequence if v0.4.2's poll tuning proves insufficient
  (IT whitelist → Gitee message plane → CF Workers + KV → self-hosted
  ntfy on a reported-in domain).

### `VERSION` / `DESIGN.md` / `README.md`

- `VERSION`: `v0.4.1 → v0.4.2`.
- `DESIGN.md` "Polling cadence" section rewritten to reflect v0.4.2
  numbers (base 5, growth 1.5, floor 1.0, cap 300 standalone); a new
  bullet in "Known MVP trade-offs" names the anti-beacon motivation
  and the Gitee-default swap.
- `README.md` message-plane description updated (`base 1s cadence,
  cap timeout/2` → `base 5s, cap 300s standalone`); Configuration
  section's Ntfy block updated (`poll_base 1.0 → 5.0`, growth/floor,
  new `ntfy_poll_cap_seconds`).

## Deliberate non-choices

- **No new envelope field**. Truncation marker still rides in-band
  in `stdout_tail` / `stderr_tail`; operators who want a first-class
  `truncated: true` boolean can request it in a follow-up.
- **No message-plane switch to Gitee**. Technically feasible — Gitee
  has a REST API comparable to GitHub's — but it would require a
  full `gitee_transport.py` and a second dedup / seed / poll state
  machine to parallel `ntfy_transport.py`. Deferred until v0.4.2
  observation data shows the poll tuning is not enough.
- **No SSE / long-poll migration**. A single-user long-lived
  connection is a stronger beacon fingerprint than a widely-jittered
  poll, not a weaker one; deliberately not introduced.
- **No submitter-side poll tuning**. Submitter cadence is bounded by
  user-facing latency (`result_timeout_seconds / 2`) so its interval
  stays in the 5–150s range for the default 300s task timeout. A
  submitter running for minutes at peak cadence is not the trigger
  surface we are trying to shrink.
- **No retro-migration of bootstrapped clones**. Operators who
  already bootstrapped `agent_forward` against GitHub keep that
  remote; only fresh `tools/bootstrap_repos.py` runs will use Gitee
  as the new default.

## Verification

- `python3 -m unittest discover tests`: **88 passing, 0 failing**.
  No test net-change (tests already supply their own `NtfyConfig`
  with fast base/growth/floor values that are insensitive to
  `Settings` defaults).
- Jitter saturation trajectory recomputed for v0.4.2 defaults
  (base=5, growth=1.5, floor=1.0, cap=300 so cap_jitter=295):
  miss counts → jitter_max {1, 2.5, 4.75, 8.13, 13.19, 20.78,
  32.17, 49.26, 74.89, 113.33, 170.99, 257.49, 295}. Saturates in
  ~13 consecutive empty polls (~20 min wall clock of perfectly
  idle executor); before then polls still happen, just at ramping
  intervals. At saturation each poll is an approximately uniform
  draw over `[5, 300]s`; mean ≈ 152s → ≈23 GETs/hour, down from
  v0.4.1's ≈47/hour.

- Manual verification of the `ntfy_poll_cap_seconds` override path:
  setting `Settings(ntfy_poll_cap_seconds=60)` and instantiating
  `Executor()` then reading back via `_bump_jitter` bounds confirms
  the cap propagates through `run_loop → poll_loop`.

## Residual risk

- **v0.4.2 is a probe, not a fix**. The single-user novelty axis of
  the gateway's heuristic is not addressable by any polling change.
  If the next 1–2 days of observation show the "isolated for hours"
  cycle still recurring at comparable frequency, the next step is
  not "tune polling more" — it's one of the §9 escalations. Do not
  interpret "it held for 36 hours" as a cure until at least a week
  of clean observation.
- **Gitee availability outside China**. Operators running the
  submitter from non-CN networks will see slower Gitee access than
  GitHub; override via `AET_FORWARD_REMOTE=https://github.com/…`
  for those hosts. Not automated.
- **Bootstrapping flow**. `tools/bootstrap_repos.py` now clones
  from Gitee by default; the bootstrap script's error messages still
  read "github" in places. Not user-facing during normal operation
  but worth cleaning on the next doc pass.
- **Submitter cadence unchanged**. If a corporate gateway learns to
  isolate ntfy.sh based on the submitter's cadence alone (rarely
  the case since submits are human-paced), this release buys
  nothing on that axis.

## Verification targets for v0.4.3 / next

- 1–2 days of natural run under the Huawei-HK proxy with the new
  polling. Record: (a) whether the "ntfy dark for hours" cycle
  recurs, (b) the interval between events if yes, (c) whether
  Gitee push/pull shows the expected latency improvement on
  `submit_files.py`.
- If recurrence persists: prototype `gitee_transport.py` as an
  alternate message plane. Envelope shape unchanged; only the
  transport module and a `Settings.message_plane` switch would be
  new. Tracked in §9.
- Consider filing an IT whitelist request for `ntfy.sh` regardless.
  The polling tune reduces the trigger surface but does not take
  the legitimate-business-need argument off the table.
