# Review v0.1

## Scope

- rework executor into dispatcher + worker + single git-writer model
- restore legacy-style submitter CLI output and timeout wording
- document single-executor steady-state semantics and separate-clone deployment

## Findings

- No blocking findings remain for `v0.1`.
- The biggest runtime mismatch from `agent_forward` was task lifecycle shape:
  - current repo had become one-task blocking after ACK
  - this version instead uses durable ACK, then one async worker owning `execute -> finalize`
- Backward writes are serialized through one writer clone so finalize does not race on one git worktree.
- User-facing submitter behavior now also matches legacy much more closely:
  - preview first
  - `SUBMITTED command_id=...`
  - timeout wording
  - stdout/stderr tail replay
- Protocol storage remains dual-repo and still uses `task_id` internally; `command_id` is now a user-facing alias in CLI/log output.
- Steady-state dispatch is explicitly single-executor: startup recovers backward once, then normal scans sync only forward.

## Residual Risk

- `ack only` is still possible if the process is interrupted after ACK becomes durable but before any final/stale result becomes durable.
- `stale` intentionally allows the local process to outlive the protocol deadline; cleanup is local-only after that point.
- Live burst results still depend on remote executor throughput and remote git/network health.

## Decision

Release `v0.1`.
