# Review v0.4.1

## Scope

Two tightly-scoped executor-side fallbacks for a real operational
problem: the relay host running the executor is behind a VPN whose
packet audit silently drops HTTP bodies above some threshold in the
80–100 KB range. Before v0.4.1, two failure modes went entirely
undetected:

1. **Oversize backward envelopes**. `stdout_tail` and `stderr_tail` are
   capped at 4000 *characters*, not bytes. A subprocess that prints
   many NULs or CJK characters can push the JSON-encoded envelope into
   the 24–40 KB range once JSON escapes (`\u0000`) and UTF-8 inflation
   (3–4 bytes per CJK glyph) compound. Rare but real.
2. **Infinite retry**. `publish_forever` retried forever. If the audit
   blackhole persisted, the worker thread spun on exponential backoff
   indefinitely, well past the submitter's timeout. No observable
   failure signal, just an eternally pending task.

Both are now closed. No new wire format, no receiver-side changes, no
attempt at HTTP-level "this error means audit blocked vs. normal
flake" — the surface for that heuristic is too small.

## Findings

### `agent_exec_tunnel/ntfy_transport.py` — `publish_forever` deadline

Added `deadline_monotonic: float | None = None` kwarg. Semantics:

- `None` (default, backward-compat): classic infinite retry.
- Not-None: returns `False` as soon as `time.monotonic() >= deadline`,
  both *before* attempting a publish (fast path on a past-deadline) and
  *after* an error before sleeping (don't oversleep the budget on a
  long Retry-After).

One log line each on the two deadline-exit paths, including the last
HTTP error text when the giveup follows a failed attempt.

### `agent_exec_tunnel/executor.py` — `_truncate_result_envelope` + wiring

New module-level helper (`_truncate_result_envelope`, `_envelope_size`).
Behavior:

- Noop when JSON-encoded envelope ≤ budget (returns original dict by
  reference; caller can rely on identity).
- Otherwise: replaces both `stdout_tail` and `stderr_tail` with a
  `[truncated by executor: original NB, envelope wire budget NB]` note,
  then measures remaining budget (after note-only envelope) minus a
  256 B slack for JSON escape surprises, divides by 2, and prepends the
  note to the *last* N characters of each original tail. If JSON
  escapes STILL push us over (degenerate case of all-control-character
  input), fall back to notes-only — guaranteed to fit because the
  notes-only envelope was measurable.

The "tail of the tail" choice is deliberate: subprocess epilogue
(`...segfault`, `exit 2`, stack traces) is more diagnostic than its
prologue. Consumers who want a specific prefix should arrange
out-of-band transport via `submit_files.py`.

`_publish_result(result, task_timeout_seconds)` and
`_publish_ack(task_id, task_timeout_seconds)` both gained the timeout
parameter. They derive a deadline (`time.monotonic() + task_timeout`)
and pass it to `publish_forever`. Callers fetch the timeout from the
task envelope with a fallback to `settings.default_timeout_seconds` so
the `_publish_envelope_failure` path (envelope rejected, no valid
timeout) still gets a bounded retry rather than falling back to the
classic infinite loop.

Retry budget choice: **full** `task.timeout_seconds`, no deduction for
subprocess wall time. User was explicit on this: don't do
`retry_timeout = task_timeout - command_time`. Publish is a distinct
failure mode from execution; they get independent shots at their
shared budget.

### `agent_exec_tunnel/config.py`

New field `ntfy_result_wire_budget_bytes: int = 60_000`, env-override
via `AET_NTFY_RESULT_WIRE_BUDGET_BYTES`. 60 KB leaves ~20–40 KB of
HTTP/TLS framing margin under typical VPN audit caps. Operators who
know their audit is tighter can drop it; operators with a bigger cap
can raise it.

## Deliberate non-choices

- **No change to `TailBuffer.limit`** (still 4000 characters, not bytes).
  Fixing that is a distinct refactor with its own compat story; the wire
  budget catches the downstream problem regardless.
- **No incoming-side guard** (executor's `poll_since` response could also
  hit the audit cap if many envelopes stack). `ntfy_poll_since` was
  already tightened from 30m → 10m by the user in a pre-v0.4.1 commit,
  which bounds the likely aggregate. A full fix would require `since=<id>`
  incremental polling; not urgent.
- **No attempt to distinguish audit-rejection from network flake at the
  HTTP error level**. `URLError("Connection reset by peer")`,
  `ConnectionResetError`, and `TimeoutError` look identical whether the
  cause is an audit drop or genuine transient flakiness. The two
  countermeasures (size guard pre-publish + bounded retry) handle both
  paths uniformly.
- **No Basic/Bearer selector for the truncation note format**. The note
  is plain English; consumers can regex on `"[truncated by executor:"`
  to detect.

## Verification

- `python3 -m unittest discover tests`: **88 passing**.
- Manual spot-check (wg-sg healthy, ntfy reachable):
  ```bash
  AET_NTFY_RESULT_WIRE_BUDGET_BYTES=5000 python3 executor/run_executor.py
  python3 submitter/submit.py --timeout-seconds 15 'python3 -c "print(\"X\"*50000)"'
  # submitter receives status=done with stdout_tail starting
  # "[truncated by executor: original 50001B, envelope wire budget 5000B]"
  ```
  Not run in this sandbox but the unit tests cover the same path.
- Manual retry-budget spot-check: set `AET_NTFY_TOKEN` to an invalid
  value so every publish 401s, submit a task with `--timeout-seconds 10`.
  Executor log should show `publish_forever gave up … reason=deadline`
  within ~10 s rather than spinning forever.

## Residual risk

- **Incoming GET response aggregate**. Still theoretically possible for
  ~100+ envelopes in the 10m replay window to overflow. Requires a
  task rate of ~1/5s sustained; not realistic for this tunnel. Tracked
  as v0.4.2 candidate.
- **`TailBuffer.limit` char-vs-byte confusion**. The budget guard
  compensates after the fact. If an operator wants *exact* `stdout_tail`
  bytes they need to wait for the TailBuffer refactor.
- **Truncation note pushes small oversize to still-oversize**. If the
  original envelope was 61_000 bytes with a 60_000 budget, the
  truncation note adds ~100 B and we may round-trip through the
  fallback notes-only path. Acceptable — notes-only is still
  informative and always fits.
- **Retry budget = full task timeout** means a publish that ties up the
  worker for the entire timeout still happens. Under a task timeout of
  300 s that's 5 minutes of wasted worker time. Could be shortened with
  a separate publish-timeout knob if operators complain; for now the
  simplicity of "reuse the timeout the user already set" wins.

## Verification targets for v0.4.2 / next

- Real-network test against an actually-audited relay (the user's SG
  host) to confirm truncation fires and the deadline elides the stuck
  worker.
- Consider an aggregate-response guard (either `since=<id>` incremental
  polling or a hard cap on `poll_since`).
- Consider surfacing the truncation marker as a distinct envelope
  field (`truncated: true`) instead of in-band prefix, so UIs can
  render a "content cropped" badge.
