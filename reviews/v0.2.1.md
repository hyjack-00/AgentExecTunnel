# Review v0.2.1

## Scope

Small follow-up release after v0.2. Four concrete asks:

1. Re-introduce an **ACK envelope** on the backward topic to narrow the
   mid-task-crash re-run window v0.2 documented as an MVP trade-off.
2. Shorten the **ntfy replay window** from 2h to 30m so per-poll
   payloads stay small and the window-edge ambiguity goes away.
3. Add a **boundary guard**: when the main poll thread sees a task
   envelope whose `created_at + timeout_seconds` is already in the
   past, silently skip instead of dispatching — closes the
   rate-limit-at-window-edge case.
4. Make `seen_ids` TTL-pruned so a long-running executor doesn't hoard
   every task_id it ever saw.

Plus one operational item:

5. Use `truststore.inject_into_ssl()` so ntfy HTTPS talks to the host's
   system CA store, not Python's bundled certifi. Soft import; falls
   back to urllib's default if `truststore` isn't installed.

## Findings

### ACK envelope

- `agent_exec_tunnel/protocol.py`: new `AckRecord` dataclass, `to_envelope()` emits `kind:"ack"` with `task_id`, `executor_id`, `ack_at`.
- `agent_exec_tunnel/executor.py::_publish_ack`: new worker-side helper wraps `publish_forever` — the ACK POST blocks the worker thread until ntfy accepts, so the main poll thread is never stalled.
- `_run_task_worker` now publishes ACK **before** running `subprocess.Popen`. If the executor is stopped during the ACK retry loop, the worker exits cleanly and releases `running_tasks` without side effects.
- Dedup bootstrap on restart (`seed_seen_ids`) continues to pull every envelope from the backward topic — now both ACK and result envelopes register. A crashed-mid-task restart therefore sees the prior ACK and will not re-dispatch.

### 30-minute replay window

- `Settings.ntfy_poll_since` default changed from `"2h"` to `"30m"`.
- Per-poll response size drops ~4× on busy topics.
- The prior 2h-boundary ambiguity ("is this task genuinely orphaned, or is the window just scrolling past its ACK?") no longer arises: any task envelope still visible at 30min with a declared 5-minute timeout is by definition past its deadline, and the boundary guard handles it.

### Boundary guard

- `Executor._is_expired(task)`: computes `(now - created_at) > timeout_seconds` robustly against malformed or missing fields.
- `_handle_task_envelope` calls it before the dedup check; expired envelopes are `_mark_seen`-ed and logged, never dispatched. Submitter for such a task has long since timed out anyway.

### seen_ids scalability

- `Executor.seen_ids` changed from `set[str]` to `dict[str, float]` (task_id → monotonic insert time).
- `_mark_seen` and `_is_seen` unified around the dict. `_maybe_prune_seen_ids` runs at most once per minute from the hot dedup path and drops entries older than `Settings.seen_ids_ttl_seconds` (default 3600s = 2× the poll window).
- A realistic 30-min window carries at most `rate × 1800s` entries (1 task/sec → 1800 entries → ~90KB). With TTL pruning the long-term footprint doesn't grow further.

### truststore

- `agent_exec_tunnel/ntfy_transport.py` top-of-module soft imports `truststore` and calls `inject_into_ssl()`. Guarded by `try/except ImportError` so environments without the package still work.
- `truststore.inject_into_ssl()` patches `ssl.SSLContext` globally, so every subsequent `urllib.request.urlopen` in this process uses system CAs.

### Submitter-side consequence

- New `match_kind` parameter on `ntfy_transport.wait_for` plus matching callers in `submitter.wait_for_result` and `submitter/_submit_common._poll_for_result`. Without this, the submitter would see the intermediate ACK envelope (same `task_id` as the eventual result) and treat its absent `status` field as `"failed"`. Caught by live smoke test — the submitter was exiting 1 with an empty payload.

## Verification

- `python3 -m unittest discover tests`: **44 passing, 0 failing**. New tests:
  - `HandleTaskEnvelopeTests::test_valid_task_publishes_ack_then_result` — asserts `_kinds(publish) == ["ack", "result"]`, in order.
  - `HandleTaskEnvelopeTests::test_expired_envelope_is_dropped_without_any_publish` — expired `created_at` + short `timeout_seconds` yields zero publishes and a seen_ids mark.
  - `HandleTaskEnvelopeTests::test_missing_*_produces_failed_result_without_ack` — validation failures still produce a single `result` publish (no ACK, because the worker never spawns).
  - `PublishForeverRetryTests::test_worker_awaits_publish_forever_for_both_ack_and_result` — order and count check.
  - `SeenIdsTtlPruneTests::test_old_entries_are_pruned_on_lazy_check` — TTL prune fires from `_is_seen`.
  - `AckEnvelopeTests::test_ack_envelope_carries_executor_id_and_task_id` — ACK shape check.
- Live E2E on real ntfy.sh: ntfy.sh's anonymous tier rate-limited our testing topic (HTTP 429). Observed behavior: `publish_forever` correctly backed off, honored `Retry-After`, and eventually recovered after 13 attempts; executor log line `ntfy publish_forever recovered topic=agent-backward-285 attempts=13` + `finalize … status=done exit=0 published=True`. The submitter gave up at its own deadline (30s + 15s grace); the executor finalized on its own schedule regardless.

## Residual risk

- The ACK window is narrowed but not zero: executor crash between main-thread claim and ACK publish-accept leaves no trace, so a restart within 30min will re-dispatch. This is inherent without on-disk pre-ACK state (explicitly out of scope for MVP).
- Under sustained rate-limit conditions (ntfy.sh anonymous tier), publish_forever blocks workers and the poll loop both experience 429. The executor absorbs this correctly; the submitter's deadline is still governed by `timeout + grace`. Production setups should use a private ntfy or an authenticated topic to avoid this.
- `match_kind` must stay wired at every caller that polls the backward topic. A future change introducing a third envelope kind (heartbeat? status update?) would need the same consideration.
