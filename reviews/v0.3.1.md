# Review v0.3.1

## Scope

Hotfix release after v0.3 — triggered by an operator report that `python3 submitter/submit_gitbash.py 'ls'` fails on a **Windows executor** with `'ls' 不是内部或外部命令` (Chinese Windows cmd.exe's "command not found"), because the executor's `subprocess.Popen(shell=True)` on Windows invokes `cmd.exe /c <command>` — and `cmd.exe` cannot execute bash-only commands like `ls`.

Root cause: the v0.3 `submit_gitbash.py` (and `submit_powershell.py`) submitted the **raw payload** as `task.command`, while the preview displayed the `git bash -c <payload>` wrap. Preview was correct about intent, envelope wire was not. On a Linux executor the raw payload happens to work because `/bin/sh -c <payload>` is bash-compatible enough. On a Windows executor it fails because cmd.exe is not a POSIX shell.

Decision (from user): the four `submit_{gitbash,powershell}[_ssh].py` CLIs are **Windows-executor-only**; they now ship the actually-wrapped command. A new `submit_bash.py` CLI targets **Linux executors**.

## Findings

### Align preview with wire — Windows-executor submitters now submit wrapped commands

- `submitter/submit_gitbash.py`: now calls `render_gitbash_relay_command(payload)` and submits the first return value (`"<git bash>" -c <payload>`) rather than raw `payload`. Preview unchanged — still shows the same Windows cmdline plus the legacy second line — but **now the envelope wire matches**.
- `submitter/submit_gitbash_ssh.py`: previously submitted `relay_script` (bare `ssh HOST "bash -c \"$(...)\""` — only runnable on a Linux executor). Now submits `command` (`"<git bash>" -c <relay_script>`), which is the path that actually works on Windows executor: cmd.exe forwards the whole argv to bash.exe, which parses the ssh base64 trampoline correctly. Payload round-trip still base64-protected end-to-end.
- `submitter/submit_powershell.py`: previously submitted raw payload; now submits `"powershell.exe -EncodedCommand <b64-utf16>"` (the first return value of `render_relay_command`). cmd.exe /c starts powershell with the UTF-16 base64 script.
- `submitter/submit_powershell_ssh.py`: already correct in v0.3 (submits `powershell_cmd`); no change.

### New: `submitter/submit_bash.py` — Linux executor

- Linux executor's `subprocess.Popen(shell=True)` is `/bin/sh -c <command>` which is a POSIX shell. No wrapping needed.
- `submit_bash.py` submits the raw payload. Own lightweight `write_bash_relay_preview` prints `-> /bin/sh -c <payload!r>` + `  -> <payload>` to match the existing preview aesthetic.
- Deliberately minimal: no ssh variant in v0.3.1 — Linux users who need ssh can hand-write `submit_bash.py 'ssh HOST "..."'`. If that becomes painful, a `submit_bash_ssh.py` would reuse the shell-neutral `ssh HOST "bash -c \"$(...)\""` that `render_gitbash_ssh_command` already produces (just without the `"<git bash>" -c` outer wrap).

### Test additions

- `tests/test_cli_entrypoints.py`:
  - `test_submit_gitbash_main_submits_wrapped_windows_cmdline` — **rewritten** to assert the submitted command contains `bash.exe`, `-c`, and the user payload (so it actually goes through git bash on Windows executor).
  - `test_submit_gitbash_ssh_main_submits_wrapped_windows_cmdline` — **rewritten** to assert outer `bash.exe` wrap plus the inner `ssh H20` plus the `base64 -d` trampoline.
  - `test_submit_powershell_main_submits_wrapped_powershell_cmdline` — **new** — asserts `powershell.exe -EncodedCommand` in the submitted command.
  - `test_submit_bash_main_submits_raw_payload` — **new** — asserts Linux path still submits unwrapped.

## Still deferred to post-v0.3.1

All of these were flagged by the v0.3 post-tag review (codex + subagent) and are intentionally NOT in v0.3.1:

- **Remote `base64` missing** → `bash -c "$(empty)"` exits 0 silent success (HIGH). Need a guard like `set -e` inside the trampoline or a decoded-empty check.
- **`$(...)` strips trailing newlines from decoded payload** (HIGH, minor in practice).
- **ARG_MAX pre-flight** (HIGH). Payloads > ~95KB will fail late with an opaque `Argument list too long`.
- **PowerShell ssh path not base64-protected** — `submit_powershell_ssh.py` still routes through `render_ssh_command`'s `--% wrapped_target` form. Quoting-heavy payloads remain fragile on the PowerShell ssh path until `render_ssh_command` grows a base64 trampoline.
- **Host option injection** — hosts starting with `-` (e.g. `-oProxyCommand=...`) slip through `shlex.quote` because `shlex.quote("-x")` returns `-x` literally. Need an allowlist or `ssh -- HOST`.
- **`poll_since` default drift** — `NtfyConfig` class default still `"2h"`; Settings default is `"30m"`; README/DESIGN mention both. Needs a sweep.
- **DESIGN.md envelope examples version strings** — ACK example says `v0.2.1`, result says `v0.2`. Update to `v0.3.1`.
- **`AET_SHOW_WIRE=1` debug env** — a 4th preview line dumping the actual envelope `command` would close the "preview ≠ wire" debug gap that this release only partially narrowed.
- **Integration test through fake-ssh shim** — current base64 tests do regex+b64 round-trip only, never actually exec through `/bin/sh -c <relay>`.

## Verification

- `python3 -m unittest discover tests`: **59 passing**.
- Manual trace of `submit_gitbash.py 'ls'`: submitted command is now `"C:\Program Files\Git\bin\bash.exe" -c "ls"`. Windows executor's `cmd.exe /c <that>` → starts `bash.exe` with `-c ls` → bash runs `ls`. Matches operator expectation.
- Preview output for all four Windows-executor CLIs unchanged from v0.3 (still human-readable, still shows intent). `submit_bash.py` preview is `-> /bin/sh -c 'payload'` + `  -> payload`.
- No live ntfy.sh E2E rerun — rate-limit still in effect from v0.2.1/v0.3 testing; unit tests cover the submitter-side wiring completely.

## Release verdict

**Ship v0.3.1.** It's a contained hotfix that unblocks Windows-executor users who hit the `'ls' 不是内部或外部命令` symptom and adds the missing Linux CLI. Remaining review items from v0.3 post-tag are tracked in PLAN.md §8.2's "deferred" list (they existed before v0.3.1 and are unchanged).
