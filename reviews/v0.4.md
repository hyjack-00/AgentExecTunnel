# Review v0.4

## Scope

`submit_files.py` becomes synchronous and verified. Previously it was fire-and-forget: local stage + `git_commit_push` and that was it — no signal that any executor ever pulled, no check that the file actually landed on the remote side. v0.4 turns the CLI into a blocking primitive: it pushes, then dispatches a rendered bash command over ntfy that makes the executor pull + verify, and waits for the result with distinct exit codes for each failure mode.

No executor changes — the new remote logic is rendered client-side in `submit_files.py::_render_remote_verify_command` and shipped as the ordinary `envelope["command"]` string. The executor still does `bash -c <command>` with no special case for file uploads.

## Findings

### `submitter/submit_files.py` — full rewrite

**New flow**:

1. `_sync_forward_root(forward_root)` — best-effort `git_sync` so the namespace-uniqueness check reflects remote state. Failure here is a warning, not fatal (we still try to push; push's rebase loop absorbs conflicts).

2. **Namespace uniqueness**. If `agent_forward/files/<name>/` already exists locally, refuse up front with a clear message: "namespaces are one-shot — pick a different --name". This closes the "operator re-ran the command and overwrote their previous upload" failure mode without a confirmation prompt.

3. `copy_tree_or_file(src, dst)` — unchanged.

4. `_push_with_retry(forward_root, message)` — up to 3 attempts with 15 s between retries. Each attempt runs `git_commit_push(max_attempts=8)` (inner rebase loop, bounded small so each outer attempt is responsive). Total budget ≤ ~45 s wall clock for git push + network time. Final failure exits **1** with a `please re-run` hint quoting the exact command line to retry.

5. `_render_remote_verify_command(name, filename)` — generates a single-line bash command:

   ```bash
   FORWARD_ROOT="${AET_FORWARD_ROOT:-agent_forward}" && \
   cd "$FORWARD_ROOT" || { echo "forward_root not found: $FORWARD_ROOT" >&2; exit 10; } && \
   ( git fetch --quiet origin main && git reset --quiet --hard origin/main \
     || ( sleep 15 && git fetch --quiet origin main && git reset --quiet --hard origin/main ) \
     || ( sleep 15 && git fetch --quiet origin main && git reset --quiet --hard origin/main ) \
     || { echo "git pull failed after 3 attempts" >&2; exit 11; } ) && \
   ( [ -e 'files/<name>/<filename>' ] \
     && echo "VERIFY_OK namespace=<name> path=..." \
     || { echo "VERIFY_MISSING namespace=<name> path=..." >&2; exit 12; } )
   ```

   Distinct exit codes (10 / 11 / 12) let the submitter disambiguate **forward_root missing** vs **pull kept failing** vs **pull OK but file absent** without parsing tail text. `shlex.quote` protects the namespace and filename from shell metacharacters.

6. `_run_remote_verify(namespace, verify_cmd)` — publishes the task with `timeout_seconds=120` (covers `3×15 s` sleep + actual pull time) and blocks on `wait_for_result`. Returns `(exit_code, stdout_tail, stderr_tail)`; `exit_code is None` signals the task never completed cleanly (ntfy publish failed, task stale, submitter-side timeout). The caller uses shape, not envelope status, to branch.

7. **Result interpretation** (three exit-code branches at the submitter process level):
   - exit 0 → `VERIFIED namespace=<name>`, process exits 0.
   - exit_code == 12 or `VERIFY_MISSING` in stderr_tail → upload + pull OK but file absent; exit **3** with hint to retry with a new `--name`.
   - any other non-zero → pull never succeeded (or task stale); exit **2** and print the exact bash command to run manually on the executor host: `bash -c <shlex.quoted verify_cmd>`.

**Error messaging**. All three failure exits print to stderr with concrete guidance: "please re-run `python3 submitter/submit_files.py --name X --src Y`" for exit 1; "to finish verification manually, run this on the executor host: `bash -c '<verify_cmd>'`" for exit 2; "unknown cause — please retry the upload with a new --name" for exit 3. The manual-retry hint is re-runnable verbatim.

### `tests/test_cli_entrypoints.py` — four new tests

- `test_submit_files_main_uploads_and_verifies_remotely`: happy path. Asserts `git_sync` called, `copy_tree_or_file` dst = `files/demo/payload.txt`, `git_commit_push` called with `max_attempts=8`, `publish_task` receives a verify command containing the relative path, three `git fetch` retries, and `VERIFY_OK`/`VERIFY_MISSING` tags, `AET_FORWARD_ROOT` env reference.
- `test_submit_files_rejects_reused_namespace`: pre-creates `files/demo/`, expects `SystemExit("…already in use…")` and `publish_task` never called.
- `test_submit_files_reports_file_missing_after_pull`: mocks `wait_for_result` to return `exit_code=12` with `VERIFY_MISSING` in stderr_tail. Asserts exit 3 and "absent from the pulled tree" in stderr.
- `test_submit_files_reports_remote_pull_failed_with_manual_hint`: mocks `wait_for_result` to return `exit_code=11` (pull failed). Asserts exit 2 and the manual-retry hint in stderr.
- `test_submit_files_reports_local_upload_failure_with_rerun_hint`: mocks `git_commit_push` to raise `CalledProcessError`. Patches `time.sleep` so the test doesn't wait 30 s. Asserts exit 1, `please re-run` hint, `publish_task` never called.

Full suite: **77 passing, 0 failing** (was 73 in v0.3.4; +4 net).

### End-to-end manual verification (against a real git repo)

- Built a bare origin + a populated agent_forward locally, pushed, deleted local tree, ran the rendered verify command with `AET_FORWARD_ROOT` pointing at the empty agent_forward. **Got `VERIFY_OK` + exit 0**, confirming the pull reconstitutes the file and the existence check succeeds.
- Built a bare origin that does NOT contain the namespace path. Ran the rendered verify command. **Got `VERIFY_MISSING` + exit 12**.
- Pointed `AET_FORWARD_ROOT` at a nonexistent directory. **Got `forward_root not found` + exit 10**.

All three outcomes match the submitter's branching logic exactly.

## Deliberate non-choices

- **No streaming progress during git push or git fetch**. The submitter prints `UPLOADED` after push and `SUBMITTED verify_task_id=…` before blocking; in between, `git_commit_push`'s quiet output is captured. For a workflow that can take ~60 s on a busy network, this is acceptable; adding progress would require surfacing stderr from the rebase loop which is noisy. Park.
- **Inner rebase retry capped at 8**, not 64 (the module default). Keeps each outer 15 s cycle responsive. An operator with real concurrent-submitter contention will want to bump this — no config knob yet.
- **Namespace uniqueness checked locally, not remotely**. A racy concurrent submitter could locally pass the check but lose the push race. The push-with-retry loop handles that by failing the outer 3×15 s loop; the operator sees exit 1 with "please re-run". Not ideal but acceptable for single-submitter workflows.
- **120 s task timeout hard-coded**. Covers 3×15 s retry + realistic pull. Not a `Settings` knob yet.
- **No "force overwrite" flag**. Namespaces are one-shot by design; if the operator wants to re-upload, they pick a new name. A force flag would reintroduce the silent-overwrite failure mode we just removed.
- **Remote command uses `git reset --hard origin/main`, not `git pull --rebase`**. `reset --hard` is idempotent and immune to local commits that might linger on the executor's repo from some previous operation. The executor's forward_root is not meant to have local edits.

## Residual risk

- **`AET_FORWARD_ROOT` fallback assumes the executor's cwd is the repo root**. If the operator starts the executor from `/home/user/` while `agent_forward` lives under `/home/user/AgentExecTunnel/agent_forward`, the fallback path `agent_forward` resolves wrong. Workaround: export `AET_FORWARD_ROOT` at executor-launch time. Documented in README.
- **12 / 11 / 10 exit codes collide with nothing by convention, but a user who reads `exit_code=127` as "not found" might be surprised**. These live inside the verify envelope, not the submitter's own exit code; distinct enough.
- **`wait_for_result` TimeoutError is caught** and mapped to exit 2 with manual-retry hint — the submitter never surfaces the TimeoutError traceback. Acceptable UX, but if an operator expects a stack trace for debugging, they won't get one (stderr message is still present).
- **The verify command is ~600 bytes of bash** — well under the 100 KB ARG_MAX limit introduced in v0.3.4, but it IS subject to that limit, so if an operator later wants a much larger verify (e.g., hashing the file), they'll need to stay under the limit or upload a verifier script via `submit_files.py` first (which is a chicken-and-egg problem for the first use).

## Verification targets for v0.5 / next cycle

- Live `submit_files.py` against a real remote executor with a valid `agent_forward` clone on both sides, to observe the `UPLOADED` → `SUBMITTED verify_task_id=` → `VERIFIED` round-trip wall time.
- Consider a `--timeout-seconds` CLI flag (and plumb it into the `publish_task` call) for operators who want to tune the 120 s budget.
- Consider whether the manual-retry hint should include a reminder about `AET_FORWARD_ROOT` for operators whose executor cwd isn't the repo root.
- Consider adding a hash-based verification (compute sha256 on the submitter, echo on the executor, compare) for operators who want bit-integrity not just presence.
