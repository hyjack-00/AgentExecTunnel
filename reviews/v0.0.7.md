# Review v0.0.7

## Scope

- correct the design description of task lifecycle and executor behavior
- explicitly separate:
  - durable ACK semantics
  - final-result polling semantics
  - current blocking executor behavior
  - not-yet-implemented async active-process monitoring

## Findings

- No blocking findings remain for `v0.0.7`.
- The prior design text mixed together:
  - protocol-level task state
  - git retry behavior
  - possible async runtime shapes
- The most important clarification is that current `AgentExecTunnel` behavior is:
  - publish task
  - executor durably pushes ACK
  - executor runs one task in a blocking way
  - executor durably pushes one final result
  - submitter polls only for final result
- The current implementation does **not** yet maintain an active asynchronous task set for long-running tasks.

## Residual Risk

- `ack only` remains possible if the executor process is externally interrupted after ACK becomes durable but before final result becomes durable.
- The current one-task blocking executor model may reduce throughput during one long task or one long outage, even though it remains operationally safe for the supported single-network environment.

## Decision

Release `v0.0.7`.
